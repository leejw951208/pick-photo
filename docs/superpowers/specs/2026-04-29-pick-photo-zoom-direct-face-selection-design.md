# Pick Photo 확대 직접 얼굴 선택 UX 설계

## 목표

Pick Photo의 얼굴 검토 화면을 사용자가 사진 속 얼굴을 직접 보고 선택하는 경험으로 설계한다. 발견된 얼굴이 많거나 작게 보이는 사진에서도 사용자가 확대와 이동을 통해 원하는 얼굴만 정확히 선택하고, 선택된 얼굴만 증명사진 스타일 결과 생성 대상으로 보낼 수 있어야 한다.

## 제품 기준

이 설계는 `PRD.md`의 다음 요구와 위험을 직접 다룬다.

- 사용자는 발견된 얼굴 중 하나를 선택할 수 있어야 한다.
- 사용자는 발견된 모든 얼굴을 한 번에 선택할 수 있어야 한다.
- 선택되지 않은 얼굴에 대해서는 결과를 생성하지 않아야 한다.
- 얼굴 선택 화면은 사용자가 서로 다른 얼굴을 구분할 수 있을 만큼 명확해야 한다.
- 잘못된 얼굴 선택은 개인정보 침해와 사용자 불신으로 이어질 수 있으므로 생성 전 선택된 얼굴이 분명해야 한다.

## 비목표

- 국가별 공식 증명사진 규격 선택 UI는 포함하지 않는다.
- 세부 보정 도구, 수동 crop 편집, 배경 교체 편집 UI는 포함하지 않는다.
- 계정, 결제, 장기 보관, 삭제 정책 확정 UI는 포함하지 않는다.
- 실제 얼굴 인식 모델 품질 개선이나 AI 파이프라인 변경은 이 UX 설계의 범위가 아니다.

## 핵심 UX 방향

얼굴 선택의 기본 인터랙션은 하단 목록 선택이 아니라 원본 사진 위 직접 선택이다. 사용자는 사진 위에 표시된 얼굴 박스를 탭해 생성 대상에 포함하거나 제외한다. 하단 영역은 얼굴 목록을 반복해서 보여주는 공간이 아니라, 선택된 얼굴 수와 생성 액션을 확인하는 짧은 요약 바가 된다.

얼굴이 많거나 작게 인식된 경우 같은 화면 안에서 확대 선택 모드로 전환한다. 사용자는 사진 영역 안에서 핀치 줌과 드래그로 사진을 확대하고 이동할 수 있다. 제스처 사용이 어렵거나 정밀 조작이 필요한 경우를 위해 `-`, `+`, `맞춤`, `원본 보기` 버튼을 함께 제공한다.

## 화면 구조

### 상단 상태 영역

- 현재 단계는 `업로드`, `검토`, `생성`, `저장` 단계 표시로 보여준다.
- 얼굴 검토 단계에서는 발견된 얼굴 수를 표시한다.
- 얼굴이 많거나 작은 경우 상단 보조 배지로 현재 확대 배율을 보여준다.

### 사진 선택 영역

- 원본 사진 위에 인식된 얼굴별 박스를 표시한다.
- 각 박스는 번호를 가진다.
- 선택된 얼굴은 색상, 체크 아이콘, 선택 상태 텍스트로 함께 구분한다.
- 확인이 필요한 얼굴은 점선 박스와 `확인 필요` 상태로 표시한다.
- 터치 가능한 영역은 보이는 얼굴 박스보다 넓게 잡아 최소 44px 이상의 조작 영역을 보장한다.

### 확대 선택 모드

- 확대와 이동 제스처는 사진 선택 영역 안에서만 작동한다.
- 앱 전체 세로 스크롤과 사진 팬 제스처가 충돌하지 않도록 사진 영역이 명확한 조작 컨테이너가 된다.
- 확대 중에는 미니맵을 표시해 사용자가 전체 사진의 어느 부분을 보고 있는지 알 수 있게 한다.
- 사용자는 언제든 `원본 보기` 또는 `맞춤`으로 전체 사진 맥락으로 돌아갈 수 있다.
- 작은 얼굴 또는 얼굴 간격이 좁은 경우 확대 모드에서 선택하도록 유도한다.

### 하단 선택 요약 바

- 선택된 얼굴 수를 표시한다.
- 선택된 얼굴을 작은 스택 형태로 간단히 보여준다.
- `선택 초기화`, `원본 보기`, `선택한 얼굴 생성` 같은 짧은 액션만 제공한다.
- 하단에 얼굴 목록을 반복해서 나열하지 않는다.

## 상태 모델

### 포함

사용자가 생성 대상으로 선택한 얼굴이다. 사진 위 박스는 실선과 체크 아이콘을 함께 사용한다. 선택된 얼굴 수와 생성 버튼 상태에 즉시 반영한다.

### 제외

인식되었지만 생성 대상이 아닌 얼굴이다. 사진 위 박스는 중립 색상으로 표시하며 다시 탭하면 포함 상태가 된다.

### 확인 필요

얼굴이 작거나 흐리거나 신뢰도가 낮아 사용자가 재확인해야 하는 상태다. 점선 박스, 경고 색상, 텍스트를 함께 사용해 색상만으로 상태를 전달하지 않는다.

### 얼굴 없음

얼굴이 발견되지 않으면 사진 선택 화면 대신 실패 안내와 새 사진 선택 액션을 제공한다.

## 데이터 요구사항

정확한 구현을 위해 얼굴 검토 화면에는 다음 데이터가 필요하다.

- `boundingBox`: 원본 사진 위 얼굴 박스를 그리기 위한 좌표. 정규화 좌표 또는 원본 픽셀 좌표 중 하나로 일관되게 제공되어야 한다.
- `confidence`: 낮은 신뢰도 얼굴에 `확인 필요` 상태를 붙이기 위한 값.
- `sourcePreviewUrl`: 얼굴 선택 화면에서 표시할 원본 사진 미리보기 URL 또는 앱이 표시할 수 있는 로컬 미리보기 참조.
- `thumbnailUrl`: 선택 요약이나 접근성 보조 화면에서 사용할 수 있는 얼굴 crop 이미지. 직접 선택 UI에서는 필수가 아니지만 사용자 확인 품질을 높인다.

현재 모바일 앱과 백엔드 계약이 이 필드를 모두 제공하지 않는다면, 구현 계획에서 API 계약과 앱 상태 모델을 함께 확장해야 한다.

## 접근성 요구사항

- 각 얼굴 박스는 스크린리더에서 `얼굴 1 선택됨`, `얼굴 2 제외됨`, `얼굴 4 확인 필요`처럼 읽혀야 한다.
- 직접 탭이 어려운 사용자를 위해 얼굴 선택 순서를 스크린리더 탐색으로도 접근 가능하게 한다.
- 색상만으로 선택 상태를 전달하지 않는다.
- 모든 터치 대상은 최소 44px 이상의 조작 영역을 가진다.
- 확대, 축소, 원본 맞춤은 제스처 외 버튼으로도 수행할 수 있어야 한다.
- 큰 글자 설정에서도 선택 요약 바와 주요 액션 버튼이 겹치지 않아야 한다.

## 개인정보와 신뢰

원본 사진, 얼굴 박스, 얼굴 crop, 선택 상태, 생성 결과는 민감한 개인 정보로 취급한다. 화면은 선택된 얼굴만 생성된다는 사실을 반복해서 보여줘야 한다. 보관 기간과 삭제 방법은 아직 제품 열린 질문이므로, 확정 정책 없이 구체적인 보관 기간을 UI 문구로 약속하지 않는다.

## 오류와 예외

- 얼굴이 너무 많아 박스가 겹치면 확대 선택 모드를 우선 안내한다.
- 사용자가 얼굴을 선택하지 않고 생성을 누르면 선택 필요 상태를 표시한다.
- 생성 중 사용자가 새 사진으로 다시 시작하면 이전 선택과 결과가 새 흐름에 섞이지 않아야 한다.
- 낮은 신뢰도 얼굴을 선택할 때는 확대 확인을 유도하되 선택 자체를 막지는 않는다.
- 부분 성공 결과는 선택한 얼굴별 결과 화면에서 성공과 실패를 분리해 보여야 한다.

## 설계 결정

최종 UX 방향은 `원본 직접 선택 + 확대 선택 모드 + 하단 선택 요약`이다. 하단 얼굴 리스트를 기본 선택 수단으로 두지 않는다. 직접 선택은 사용자가 실제 사진 속 인물을 바로 고르게 하므로 잘못된 얼굴 선택 위험을 줄이고, 확대 선택은 단체 사진과 작은 얼굴의 정밀 선택 문제를 보완한다.

## 구현 전 확인할 결정

- API 응답에서 `boundingBox`와 `sourcePreviewUrl`을 어떻게 제공할지 결정해야 한다.
- 얼굴 crop 썸네일을 서버가 제공할지, 앱이 원본 미리보기에서 직접 crop할지 결정해야 한다.
- `확인 필요`의 기준이 되는 confidence 임계값을 결정해야 한다.
- 원본 사진과 얼굴 미리보기의 보관 및 만료 정책을 결정해야 한다.

## 스펙 자체 점검

- Placeholder scan: 미완성 표기나 빈 섹션은 없다.
- Internal consistency: 하단 리스트 제거, 직접 선택, 확대 선택, 선택 요약 바 구조가 전체 문서에서 일관된다.
- Scope check: 이 스펙은 얼굴 검토 UX와 필요한 데이터 계약에 집중하며, AI 모델 개선이나 공식 증명사진 규격 UI는 범위 밖으로 둔다.
- Ambiguity check: 선택 방식은 `원본 직접 선택 + 확대 선택 모드`로 확정했고, 하단 리스트는 기본 선택 수단이 아니라고 명시했다.
